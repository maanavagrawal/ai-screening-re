# Lessons

- This repo starts from an empty scaffold. Keep setup files lightweight until the actual application stack is added.
- Use `./scripts/setup.sh`, `./scripts/dev.sh`, `./scripts/test.sh`, and `./scripts/e2e.sh` as the stable entrypoints for Codex Cloud and local Codex work.
- Local Codex can use the in-app browser for live watchable UI iteration; Codex Cloud should leave logs, screenshots, commits, and PRs as evidence.
- Deployment preference changed after setup friction: target Railway for the Next.js frontend/API and Railway Postgres for persistence. Legacy provider compatibility layers should be deleted once the active path is verified, not kept indefinitely.
- Railway/Nixpacks will honor `packageManager` in `package.json`; do not declare pnpm unless `pnpm-lock.yaml` is committed. This repo uses `package-lock.json`, so force `npm ci` via `nixpacks.toml`.
- The Codex in-app browser reaches this local dev server through the LAN URL `http://10.0.0.37:3001`; `localhost`/`127.0.0.1` can be blocked in that surface.
- LAN preview is not always a secure context, so browser APIs such as `crypto.randomUUID()` need practical fallbacks in client code.
- Keep Playwright e2e deterministic: start a fresh dedicated server port, disable external AI/Twilio calls with env flags, and avoid reusing stale Next dev servers.
- Do not run `next build` at the same time as Playwright or a watched `next dev` server. Next mutates `.next`, and overlapping processes can produce stale chunk/runtime errors that disappear when build and dev are run separately.
- The Codex in-app browser is best for visible navigation, clicks, screenshots, and console checks. If text entry fails because the browser surface lacks clipboard APIs, use Playwright e2e for form-typing proof and keep browser QA focused on live visual/click verification.
- Intake screens must guard pending advancement. Without a pending ref/state, repeated Continue clicks can create overlapping `/api/intake/next` requests and stuck transitions.
- Listing cards must not use stock fallback property images. Buyer-facing media should be actual agent-provided listing media or future authorized MLS/IDX media; no-media cards should use a neutral branded panel.
- The production root `/` is agent-facing setup, not a pilot buyer page. Pilot agents belong only at their own slugs such as `/maya` and `/david`; completed setup creates the shareable `/{agentSlug}` link.
- Railway server routes should avoid browser-only runtime constructors such as `File`; validate multipart uploads structurally and keep `sharp` routes pinned to `runtime = "nodejs"`.
- Production signup must fail closed if `DATABASE_URL`/`RESEND_API_KEY` or another real auth provider is missing. The dev setup redirect belongs only in local dev or with explicit `ALLOW_DEV_AGENT_AUTH=1`.
- Do not use remote placeholder video blocks in setup screens. If media is not essential and locally reliable, remove it rather than risking a blank framed rectangle in production.
- Do not build public links from `request.url` on Railway. Use `NEXT_PUBLIC_APP_URL` first, forwarded host/proto second, and reject internal bind hosts such as `0.0.0.0`.
- Magic-link GET routes must not consume tokens. Email clients and security scanners can prefetch GET links; render a no-store confirmation page and consume only on POST.
- Keep listing import production-safe: reduce agent typing by extracting fields from user-provided captions, MLS remarks, flyer copy, notes, CSVs, or future authorized IDX/RESO feeds; do not build stealth Zillow/Redfin/MLS scraping or anti-bot evasion.
- Exact listing addresses are agent-only data until a serious showing flow. Buyer-facing pages, APIs, and buyer AI prompts should use redacted listing payloads or neighborhood-level labels so users cannot inspect network/HTML and bypass the agent.
- Twilio Verify error 21608 means the Twilio account is still in trial mode and can only text Twilio-verified recipient numbers. Surface this as an actionable setup/infrastructure error; the production fix is upgrading Twilio or verifying tester numbers, not bypassing phone verification in app code.
- Dashboard distribution templates are optional cache data. Do not eagerly generate them on leads/listings/settings pages, and tolerate stale-agent foreign-key races so cache writes never crash unrelated dashboard or setup flows.
- Setup drafts can contain empty strings from controlled inputs. Reusable server primitives such as `onboardAgent()` should normalize blank optional values to null while keeping required fields strict, so existing drafts survive deploys and users do not restart setup.
- Do not trade away the buyer experience's AI quality for speed. Keep AI where it creates visible product value (free-text extraction, match reasons, briefs) and remove only redundant per-click latency, with tests that guard against repeated or lower-quality intake screens.
- Structured buyer-intake taps should not wait on server routes. Keep deterministic next-question selection in a shared client-safe helper and reserve network calls for value-adding AI steps such as free-text extraction.
- Agent magic-link login must distinguish new setup from returning access. Never reset an existing `setup_drafts.current_step` when sending a login link; post-login routing should prefer dashboard for completed agents and resume the saved draft step for incomplete agents.
- `getCurrentAgent()` has a local dev preview fallback that returns the first pilot agent when no authenticated user exists. Do not use it to decide whether someone is actually signed in; auth-gated flows need a helper that starts from `getCurrentUserId()`.
- Setup-time APIs can run before an agent row exists. Authorize those routes with `getCurrentUserId()` and reserve `getCurrentAgent()` for post-setup dashboard flows or routes that truly require a completed agent profile.
- For Playwright forms backed by client-only state such as `useSessionId`, wait for an explicit ready marker before filling fields. Otherwise hydration can reset controlled values and make submit failures look like server bugs.
- Setup wizard e2e should wait for `data-testid="setup-ready"` before typing. The setup pages are client-heavy, and filling inputs before hydration can be reset by React's initial client state.
- If a local Next dev server serves the page shell but browser clicks do not hydrate after `next build`, treat the server as stale. Start a clean dev server on a fresh port before live UI smoke testing.
- Responsive role panels should use one reordered DOM node instead of separate hidden desktop/mobile forms. Duplicate hidden form fields can confuse Playwright strict locators and create accessibility ambiguity.
- Do not mark gstack plan/design/eng reviews complete just because the plan follows their shape. Run the actual skill workflow/logging where available, record any fallback reason such as mockup generation auth failure, and update the review report after the real run.
- For listing media, do not add extra permission-checkbox friction. Treat agent-uploaded or direct agent-provided MP4 media as intentional listing media, but do not scrape social/listing-site URLs into hosted MP4s.
- Buyer location selection must use stable area identity, not visible labels alone. Google Places can return duplicate city names such as San Ramon in different countries/states, and clicking one suggestion should not select every matching label.
- Buyer social-video match cards still use the no-media panel behind external Instagram/TikTok links; reserve footer space and use social-specific copy so external CTAs never overlap placeholder text.
- Product demo WebPs should preserve whole-screen context unless the user explicitly asks for tight crops. Use cursor movement and a coherent user-flow sequence for emphasis before resorting to zoomed-in framing.
- Product demos must not show local development overlays such as Next.js issue badges. Re-record from a clean production-like surface or mask the overlay in the demo render before delivery.
- Client-supplied `lead_id` is never enough to authorize buyer mutations or analytics side effects. Verify the lead belongs to the resolved agent and the current buyer session before logging lead-bound events, sending verification SMS, requesting showings, or recomputing lead temperature.
- Routes under `/api/internal/*` need explicit internal authentication in production. Local dev can remain ergonomic, but production should fail closed if `INTERNAL_API_SECRET` is absent.
- ATTOM and Google Places are core product providers, not nice-to-have fallbacks. Runtime feature routes and health checks should fail loudly when `ATTOM_API_KEY` or `GOOGLE_PLACES_API_KEY` is missing; keep fixture helpers explicit in tests only.
- Dashboard listing entry should make provider enrichment feel one-step: address autocomplete should offer keyboard selection, selected addresses should immediately run property lookup, and draft resets must clear stale lookup previews.
- Setup listing entry should also be address-first. Media links and caption/text autofill are secondary helpers because social posts cannot provide reliable property facts; after address selection or text extraction, reveal the common fields for confirmation instead of showing a full blank form up front.
- Setup listing blur saves must be idempotent around property lookup. If an address is already synced, blur should no-op; if focus is moving to `Lookup facts`, the lookup flow should own the commit so stale address-only saves cannot clear enrichment.
- Setup neighborhood entry should use provider-backed autocomplete biased by the agent's market, and one selected city/neighborhood is enough to continue and publish. Do not reintroduce a four-neighborhood requirement.
- Address autocomplete dropdowns that share a mobile column with a lookup/action button must take layout space on mobile and only become absolute overlays at desktop breakpoints. Otherwise the suggestions panel can intercept taps meant for the action button.
- When removing a helper/importer path, make sure the surviving manual field preserves the important derived metadata. Setup `Video URL` must infer `videoSource` from Instagram, TikTok, and mp4 URLs because buyer cards key media behavior off `video_source`.
- Dynamic React lists with removable cards and local editor state must not use array indexes as keys. Keep stable client-side keys so address/source-text/fallback state cannot move to another listing after deletion.
- Setup basics Market should use provider-backed city autocomplete, not a static datalist. Scope basics suggestions to city/locality results so agents can type any city instead of picking from a tiny preset list.
- Listing address autocomplete should not invent a manual dropdown option when Google Places has no predictions. A typed partial address can look like a selected provider result, then ATTOM rejects it; leave manual entry to the input/lookup path with clear fallback copy.
- Normalize Google address labels before ATTOM lookup: strip trailing `USA`/country text and convert full US state names to two-letter abbreviations before building ATTOM `address1`/`address2` params.
- Treat ATTOM `400` as an address-shape/manual-entry fallback, not a raw provider outage. Show actionable copy and open text/manual fallback fields while preserving true config/provider failures for missing keys or unavailable ATTOM access.
- Google Places autocomplete labels are not enough for ATTOM. When the user selects an address suggestion, pass the `placeId` through the property lookup route and resolve Place Details server-side for address components/postal code before calling ATTOM.
- Never show `No structured facts found yet.` to agents after lookup fallback. It reads like an internal parser failure; tell them to fill details manually or use text autofill.
- Setup listing cards should not stack explanatory paragraphs after a lookup. Use one compact status banner, then let the form fields do the work; keep text autofill as a compact recovery panel.
- If ATTOM returns an address match but no usable structured facts, treat the UI state as `Details needed`, not `Property facts added`.
- When a setup listing address changes, clear visible address-specific fields immediately, not only enrichment metadata. Otherwise the card can show stale beds, baths, sqft, neighborhood, price, and property type while the agent is entering the next property.
- Provider lookup application should replace provider-controlled facts rather than prefer existing field values unless explicit user override tracking exists. Error/fallback patches after a clear must also include the cleared fields so stale closure state cannot re-save the previous property.
- Removable setup listing cards must route async saves through stable card keys, not array indexes. Slow lookups or text extraction can resolve after deletion, and index-based patches can mutate the wrong listing.
- Same-address lookup retries that return manual/no-facts should preserve the agent's manually entered fields. Only clear address-specific visible fields when the address actually changes.
